# 株用語解説ショート動画 自動生成パイプライン 設計ドキュメント

## 1. 目的とスコープ

### 1.1 目的
株式投資の初心者向けに、専門用語（ローソク足、PER、PBR など）を1語ずつ30秒〜1分で解説するショート動画を、可能な限り自動で生成し、毎日1本 YouTube / TikTok / Instagram リールの3媒体へ配信する仕組みを構築する。縦型1本の動画を3媒体で使い回す。

### 1.2 スコープ
- **対象**: ローカル（Mac）で完結する自動生成パイプライン。
- **本ドキュメントの範囲**: ネタ管理 → 台本生成 → 音声合成 → 動画レンダリング → マルチプラットフォーム配信までの一気通貫設計。
- **対象外（後続検討）**: クラウド化（Vercel / Raspberry Pi への移設）、スケジューラの常駐化、収益化対応。まずローカルで1本通すことを最優先とする。

### 1.3 非機能方針
- 各段階を疎結合に保ち、間に **シーン定義（JSON）レイヤー** を噛ませてレンダラーを抽象化する。これにより将来 Remotion から別レンダラーへ差し替え可能とし、ロックインを回避する。
- 1コマンドで「ネタ選定〜MP4出力」まで通る CLI を最終形とする。アップロードのみ独立コマンドに分離（誤投稿防止）。

---

## 2. 全体アーキテクチャ

### 2.1 パイプライン全体像

```
①ネタ選定        Supabase 用語マスタ → 未使用を1語 pick
   ↓
②台本生成        Gemini Flash（無料枠・narrator-profile 流用）→ 構成テンプレに沿った台本
   ↓
③シーン定義生成   台本 → scene.json（レンダラー非依存の中間表現）
   ↓
④音声合成        VOICEVOX（ローカル）→ ナレーション WAV + タイミング情報
   ↓
⑤動画レンダリング Remotion（React）→ 字幕・図解付き MP4
   ↓
⑥配信            publisher 抽象レイヤー → 3アダプタへ分配
                   ├─ YouTube   : Data API v3 → 限定公開アップ
                   ├─ TikTok    : Content Posting API → アプリ下書きへ送信
                   └─ Instagram : Google Drive に保存 → 手動アップ
```

### 2.2 段階間のデータ受け渡し

| 段階 | 入力 | 出力 |
|------|------|------|
| ①ネタ選定 | 用語マスタ | term レコード（用語・カテゴリ） |
| ②台本生成 | term レコード | script（テキスト＋構成メタ） |
| ③シーン定義 | script | scene.json |
| ④音声合成 | scene.json の各テキスト | narration.wav ＋ 各シーンの再生秒数 |
| ⑤レンダリング | scene.json ＋ wav ＋ 秒数 | output.mp4 |
| ⑥配信 | output.mp4 ＋ メタデータ | 各媒体への配信結果（動画ID / 下書きID / ドライブリンク） |

## 3. 各コンポーネント設計

### 3.1 ①ネタ管理（Supabase）

用語マスタを Supabase に持ち、未使用のものから1語を選定する。重複防止と進捗可視化のため DB 管理とする。

**テーブル: `terms`**

| カラム | 型 | 説明 |
|--------|------|------|
| id | uuid | PK |
| term | text | 用語名（例: ローソク足） |
| reading | text | 読み仮名（TTS 用） |
| category | text | カテゴリ（チャート / 指標 / 制度 など） |
| difficulty | int | 難易度 1〜3 |
| status | text | `pending` / `generated` / `published` |
| published_at | timestamptz | 投稿日時 |
| youtube_video_id | text | YouTube 動画ID |
| tiktok_draft_id | text | TikTok 下書きID |
| drive_link | text | Instagram用 ドライブ共有リンク |

**選定ロジック**: `status = 'pending'` の中から `difficulty` 昇順・カテゴリ分散を考慮して1語 pick。初期に50〜100語を投入しておけば数ヶ月分の在庫になる。

**初期投入候補（例）**: ローソク足、陽線・陰線、PER、PBR、ROE、出来高、移動平均線、ゴールデンクロス、デッドクロス、信用倍率、配当利回り、時価総額、ストップ高・安、板、約定、etc.

### 3.2 ②台本生成（Gemini Flash・無料枠）

`movie-script-app` の narrator-profile システムをそのまま流用する。30秒〜1分（読み上げで概ね200〜350字）。

**モデル**: Gemini 2.5 Flash または Flash-Lite を使用。台本生成は短文・構成固定のためFlash系で十分。

**無料枠で回せる根拠**: 用途は1日1〜数本（=1日数リクエスト）。無料枠は Flash が10 RPM・1日250リクエスト、Flash-Lite が15 RPM・1日1,000リクエストで桁違いに余裕がある。クレジットカード不要。

**抽象化方針**: 台本生成は `script-generator` インターフェースで抽象化し、実装としてGeminiアダプタを差し込む。無料枠は予告なく改定されうる（2025年12月に50〜80%削減の実績あり）ため、将来Claude等へ差し替えてもパイプラインが壊れない構造とする。レンダラーを scene.json で抽象化したのと同じ思想。

**留意点（無料枠の制約）**
- **Proモデルは使わない**: 2026年4月以降 Pro系は有料専用。必ずFlash系で設計する。
- **学習利用**: 無料枠では入出力がGoogleのモデル改善に利用される。本用途は公開予定の株用語解説で機密性ゼロのため実害なし。
- **課金有効化の罠**: このGCPプロジェクトで課金を有効にすると無料枠が完全消滅し全リクエストが課金対象になる。無料運用を続けるなら課金を有効化しない。検証用と本番を分けたい場合はGCPプロジェクトを分離する。

**構成テンプレート（固定）**
```
1. フック   … 1文で興味を引く（「この赤と青の棒、何かわかりますか？」）
2. 定義     … 平易な言葉で一言定義
3. 具体例   … 数字や身近な例で説明
4. まとめ   … 一言で締め＋次回予告的フック（任意）
```

**出力フォーマット**: テキストをシーン単位に区切った JSON（後段でそのままシーン定義に変換できる形）。Gemini の JSON モードを利用し構造化出力を安定させる。

**免責**: 「本動画は投資助言ではなく教育目的です」を台本末尾またはテロップ用メタに必ず含める。

### 3.3 ③シーン定義レイヤー（scene.json）

レンダラー非依存の中間表現。台本をシーン配列に展開し、各シーンに表示テキスト・図解タイプ・ナレーションを持たせる。

```json
{
  "term": "ローソク足",
  "disclaimer": "本動画は投資助言ではなく教育目的です",
  "scenes": [
    {
      "id": 1,
      "type": "hook",
      "narration": "この赤と青の棒、何かわかりますか？",
      "caption": "ローソク足って何？",
      "visual": "candle_intro"
    },
    {
      "id": 2,
      "type": "definition",
      "narration": "ローソク足は…",
      "caption": "1本で4つの価格を表す",
      "visual": "candle_anatomy"
    }
  ]
}
```

- `visual` は Remotion 側のコンポーネント名にマップする（`candle_intro`, `candle_anatomy`, `cross_animation` など）。
- このレイヤーがあることで、将来 FFmpeg 版など別レンダラーへ同じ JSON を流せる。

### 3.4 ④音声合成（VOICEVOX）

- **方式**: VOICEVOX をローカルエンジンとして起動し、HTTP API（`/audio_query` → `/synthesis`）経由で WAV を生成。無料・ローカル・商用利用可。
- **入力**: 各シーンの `narration` テキスト。
- **出力**: シーンごとの WAV、または結合した1本の WAV ＋ 各シーンの再生秒数。
- **タイミング**: `audio_query` のレスポンスから各シーンの尺を算出し、シーン定義に書き戻す（字幕同期に使用）。
- **話者**: narrator-profile に対応する VOICEVOX 話者IDを固定。

### 3.5 ⑤動画レンダリング（Remotion ※確定）

React コンポーネントで動画を構成し、`npx remotion render` で MP4 出力する。

**採用理由**
- Next.js スキルが直接活き、コンポーネント資産が蓄積するほど制作が速くなる。
- ローソク足が1本ずつ形成される、移動平均線が交差する等の「動く図解」が型で書ける。教育コンテンツの価値の中核。
- Cloud9 のデザイントークン（フォント・配色）を共通化でき、ブランド統一が可能。

**留意点**
- Chromium ベースでレンダリングが重い（1本あたり数十秒〜数分）。ローカル（Mac）前提のため実害は小さい。Raspberry Pi ではレンダリングしない（将来クラウド化時はトリガー専用とする）。
- ライセンスは現状の個人・小規模規模では無料枠内。規模拡大時に要確認。

**構成設計**
- 縦型ショート（1080×1920）を基本。
- `scene.json` を props として受け取り、シーン配列を順に描画する単一 Composition。
- `visual` 名 → 図解コンポーネントのマッピングテーブルを持つ。
- 字幕は VOICEVOX の尺情報に同期して表示。
- 免責テロップを全編または冒頭/末尾に自動挿入。

**図解コンポーネント例**
- `candle_intro` … ローソク足の登場アニメ
- `candle_anatomy` … 始値・終値・高値・安値の分解図
- `cross_animation` … ゴールデン／デッドクロスの線交差
- `ratio_calc` … PER などの数式アニメーション

### 3.6 ⑥配信レイヤー（マルチプラットフォーム）

縦型1080×1920の `output.mp4` を1本作り、3媒体へ分配する。各媒体の差異を吸収するため共通の `publisher` インターフェースを定義し、媒体ごとにアダプタを実装する。

**publisher インターフェース（概念）**
```
interface Publisher {
  name: string
  publish(video: VideoFile, meta: VideoMeta): Promise<PublishResult>
}
```
`VideoMeta` はタイトル・説明・タグ・免責文を持ち、3アダプタで共有する。これにより媒体追加・差し替えが局所化される。

#### 3.6.1 YouTube アダプタ（Data API v3）
- **方式**: `videos.insert` でアップロード。初回のみ OAuth 認証、以降は refresh token で自動化。Playwright より堅牢なため API を採用。
- **到達地点**: `unlisted`（限定公開）でアップ。目視確認後に公開へ切り替え。
- **クォータ**: アップロードは1回あたり約1,600、1日の無料枠10,000。1日1本なら余裕（理論上6本まで）。課金不要。
- **投稿後処理**: 成功時に `terms.status` を `published` に更新し、`video_id`・`published_at` を記録。

#### 3.6.2 TikTok アダプタ（Content Posting API）
- **方式**: Content Posting API の **Upload(下書き)モード** を使用。動画を TikTok アプリの編集画面に送り込み、ユーザーがアプリ側で確認・編集してから投稿する。「できるとこまで自動、最後は人」の方針に合致。
- **到達地点**: アプリの下書きに動画が入った状態。最後の投稿操作のみ人手。
- **留意点**: アプリ未審査（unaudited client）状態では公開範囲が制限される。下書きモード運用なら影響は最小。Direct Post（直接公開）への移行は審査通過後に検討。

#### 3.6.3 Instagram アダプタ（Google Drive 経由）
- **方針**: プロアカウント（ビジネス/クリエイター）を持たないため、Graph API は使わず割り切る。`output.mp4` を Google Drive の指定フォルダにアップロードし、共有リンクを返すだけの薄い実装。
- **到達地点**: ドライブにファイルが保存された状態。ユーザーがDLしてアプリから手動アップ。
- **利点**: プロアカウント化・FBページ連携・公開URLホスティングといった重い前提を一切抱えずに済む。
- **将来拡張**: プロアカウント化した場合、このアダプタのみ Graph API（Content Publishing）版に差し替えれば他は無傷。

#### 3.6.4 各媒体の到達地点まとめ

| 媒体 | 方式 | 到達地点 | 最後の人手 |
|------|------|---------|-----------|
| YouTube | Data API v3 | 限定公開アップ | 公開ボタン |
| TikTok | Content Posting API（下書き） | アプリ下書きに送信 | 確認→投稿 |
| Instagram | Google Drive 保存 | ドライブにファイル保存 | DLして手動アップ |

**Playwright について**: TikTok・Instagram は自動操作の検知が厳しくアカウント制限のリスクがあるため、公式APIが使える範囲は必ずAPIを優先し、Playwright での画面自動操作は採用しない。

## 5. 実行フロー（MVP）

### 5.1 最小構成で1本通す
```
1. VOICEVOX エンジンをローカル起動
2. npm run generate          → ネタ選定〜MP4出力まで一気通貫
3. output/output.mp4 を目視確認
4. npm run publish           → 3媒体へ配信
                                （YouTube限定公開 / TikTok下書き / Driveへ保存）
5. 各媒体で最後の人手（YT公開ボタン / TikTok投稿 / IG手動アップ）
```

媒体を絞りたい場合は `npm run publish -- --only youtube` のように引数で選択可能とする。

### 5.2 日次運用（ローカル）
- 当面は手動で `generate` → 確認 → `publish` を回す。
- 安定後、`generate` を cron / launchd で毎朝自動実行し、生成物を確認してから `publish` する半自動運用へ。完全自動化（無確認投稿）はコンテンツ品質リスクを踏まえ慎重に判断する。

## 7. リスクと留意点

| リスク | 内容 | 対策 |
|--------|------|------|
| YouTube 規約 | 完全自動・大量投稿は低品質反復コンテンツと見なされうる | 1日1本・各動画に固有価値を持たせる。無確認投稿は避ける |
| 金融情報の免責 | 投資助言と誤認されるリスク | 全動画にテロップ＋説明文で「教育目的・助言ではない」を自動挿入 |
| レンダリング負荷 | Remotion は重い | ローカル（Mac）で実行。Pi ではレンダリングしない |
| 用語の正確性 | 自動生成の誤り | 初期は台本・動画を目視確認してから投稿 |
| ロックイン | レンダラー依存 | scene.json で中間表現を分離し差し替え可能に |
| Gemini無料枠の改定 | 予告なく削減/廃止されうる | script-generator で抽象化し他モデルへ差し替え可能に。課金は有効化しない |
| TikTok 審査 | 未審査クライアントは公開範囲制限 | 下書きモードで運用。Direct Post は審査通過後 |
| 配信媒体の規約 | 自動操作の検知でアカウント制限 | 公式APIを優先し Playwright 画面操作は不採用 |
